# Rails 8 Known Issues

Known bugs and workarounds for Rails 8.0.

---

## Rails 8.0

### Solid Queue MySQL Migration Fails on Tables with No Primary Key

**Symptom:** `rails db:migrate` raises `Mysql2::Error: Specified key was too long; max key length is 3072 bytes` or `can't create table` when running Solid Queue migrations on MySQL 5.7 or certain MariaDB versions.

**Cause:** Solid Queue's initial migration set uses string columns for job class names and queue names without explicit length limits. MySQL's default row format (`COMPACT`) has a key length limit lower than what the indexes require. MySQL 8.0+ with `ROW_FORMAT=DYNAMIC` handles this without issue; older versions do not.

**Affected versions:** 8.0.0–8.0.1 with MySQL < 8.0 or MariaDB < 10.3

**Status:** `fixed in 8.0.2` — migrations now specify column lengths explicitly.

**Workaround:**
Set the InnoDB row format to `DYNAMIC` before running migrations:
```sql
SET GLOBAL innodb_default_row_format = 'dynamic';
```
Or add to `config/database.yml`:
```yaml
production:
  adapter: mysql2
  encoding: utf8mb4
  charset: utf8mb4
  collation: utf8mb4_unicode_ci
  variables:
    innodb_default_row_format: dynamic
```

### Kamal 2 `deploy.yml` Generated by `rails new` Missing `proxy` Block for Non-Singleton Apps

**Symptom:** `kamal deploy` succeeds but the app is unreachable at the configured domain. Kamal reports the proxy as healthy but requests return 502.

**Cause:** The Kamal 2 config generated by `rails new --kamal` in Rails 8.0.0 omits the `proxy:` top-level block when more than one service is defined. Without `proxy:`, Kamal defaults to internal port 80, but Thruster (the default HTTP proxy) listens on 3000 inside the container.

**Affected versions:** 8.0.0 with Kamal 2.x

**Status:** `fixed in 8.0.1`

**Workaround:**
Add the `proxy` block to `config/deploy.yml`:
```yaml
proxy:
  ssl: true
  host: myapp.example.com
  app_port: 3000
```

### Propshaft Does Not Fingerprint Assets Served from `vendor/assets`

**Symptom:** Assets placed in `vendor/assets/` are served but without a digest fingerprint. Long-lived CDN or browser caches serve stale files after updates.

**Cause:** Propshaft's default load path in Rails 8 includes `app/assets` and `public/assets` but not `vendor/assets`. Files there are not processed by the digest pipeline.

**Affected versions:** 8.0.0+ with Propshaft

**Status:** `workaround only` — `vendor/assets` is officially unsupported by Propshaft; the recommendation is to move files.

**Workaround:**
Move vendor assets into `app/assets/`:
```bash
mkdir -p app/assets/vendor
mv vendor/assets/javascripts/* app/assets/vendor/
```
Or explicitly add the path in `config/application.rb`:
```ruby
config.assets.paths << Rails.root.join("vendor/assets/javascripts")
config.assets.paths << Rails.root.join("vendor/assets/stylesheets")
```
Note: Adding these paths makes Propshaft serve the files, but fingerprinting may still not apply depending on Propshaft version.

### Sprockets-to-Propshaft Migration: `asset_path` Helper Returns Wrong Path

**Symptom:** After switching from Sprockets to Propshaft, `asset_path("application.css")` returns `/assets/application.css` instead of the fingerprinted path `/assets/application-abc123.css`. Assets load but caching headers are wrong.

**Cause:** Sprockets and Propshaft resolve asset helpers differently. Propshaft's `asset_path` uses a sweep of the compiled output directory at boot. If the output directory is stale or the manifest has not been rebuilt, non-fingerprinted paths are returned.

**Affected versions:** 8.0.0 during Sprockets → Propshaft migration

**Status:** `workaround only`

**Workaround:**
1. Remove Sprockets entirely:
   ```ruby
   # Gemfile — remove these:
   # gem "sprockets-rails"
   # gem "sprockets"
   ```
2. Clear compiled assets:
   ```bash
   bin/rails assets:clobber
   bin/rails assets:precompile
   ```
3. Verify `config/application.rb` does not load `sprockets/railtie`.
4. Ensure all `javascript_include_tag` and `stylesheet_link_tag` calls do not pass `:all` (Sprockets-only option).

### Thruster TLS Termination Conflicts with Load Balancer TLS on Standard Ports

**Symptom:** HTTPS requests return `SSL_ERROR_RX_RECORD_TOO_LONG` or the browser shows "connection reset" when Thruster is deployed behind a load balancer that also does TLS termination.

**Cause:** Thruster defaults to terminating TLS itself when `SSL_DOMAIN` is set. If a load balancer upstream also terminates TLS and forwards plain HTTP to port 3000, Thruster receives plain HTTP but is configured to speak TLS, causing the handshake failure.

**Affected versions:** 8.0.0 with Thruster

**Status:** `workaround only` — intended configuration is either Thruster-terminates-TLS or LB-terminates-TLS, not both.

**Workaround:**
When the load balancer handles TLS, unset `SSL_DOMAIN` in the container environment and disable Thruster's TLS:
```yaml
# config/deploy.yml (Kamal)
env:
  clear:
    SSL_DOMAIN: ""
```
Or in the Dockerfile:
```dockerfile
ENV SSL_DOMAIN=""
```
Thruster will then proxy plain HTTP from the LB to Puma.

### `rails credentials:edit` Fails on First Run with Missing Master Key

**Symptom:** `rails credentials:edit` raises `ActiveSupport::MessageEncryptor::InvalidMessage` or exits without opening an editor on a freshly cloned repository.

**Cause:** Rails 8 splits credentials into environment-specific files (`credentials/production.yml.enc`, `credentials/staging.yml.enc`). The master key for each environment must be present in `config/credentials/{env}.key` or via `RAILS_MASTER_KEY`. A cloned repo without these keys cannot decrypt or create credentials.

**Affected versions:** 8.0.0+

**Status:** `workaround only` — by design; keys must be distributed out-of-band.

**Workaround:**
1. Obtain the key from a team member or secrets manager and write it to the correct path:
   ```bash
   echo "paste-key-here" > config/credentials/production.key
   chmod 600 config/credentials/production.key
   ```
2. Or use the `RAILS_MASTER_KEY` environment variable:
   ```bash
   RAILS_MASTER_KEY=paste-key-here bin/rails credentials:show
   ```
3. To generate a new credentials file from scratch (destructive — loses existing secrets):
   ```bash
   rm config/credentials/production.yml.enc
   bin/rails credentials:edit --environment production
   ```

### Solid Cache + Solid Queue Share Database Tables Causing Migration Conflicts

**Symptom:** `rails db:migrate` raises `Table 'solid_cache_entries' already exists` or `Table 'solid_queue_jobs' already exists` when running on a database that previously had one of the gems installed but not the other.

**Cause:** Rails 8 generators for Solid Cache and Solid Queue both write to the same SQLite database by default (`storage/production.sqlite3`). When both are present, migration version numbers can collide if the app was upgraded incrementally.

**Affected versions:** 8.0.0

**Status:** `fixed in 8.0.1` — generators now use distinct migration timestamps.

**Workaround:**
Manually rename the conflicting migration files to use non-colliding timestamps:
```bash
# List current timestamps
ls db/migrate/
# Rename the conflicting file, e.g.:
mv db/migrate/20240101000001_create_solid_cache_entries.rb \
   db/migrate/20240101000099_create_solid_cache_entries.rb
```
Then rerun `bin/rails db:migrate`.

### Active Storage `variant` Calls Fail with Vips After Upgrade from MiniMagick

**Symptom:** `ActiveStorage::InvariableError` or blank images after switching `config.active_storage.variant_processor` from `:mini_magick` to `:vips` in Rails 8.

**Cause:** Vips and MiniMagick use different operation APIs. Variant transformations written for MiniMagick (e.g., `{ resize: "100x100^" }`) are not understood by the Vips processor.

**Affected versions:** 8.0.0 when migrating from MiniMagick to Vips

**Status:** `workaround only` — both APIs are intentionally different.

**Workaround:**
Rewrite variant calls to use Vips-compatible syntax:
```ruby
# MiniMagick style (no longer works with Vips)
blob.variant(resize: "100x100^", gravity: "Center", extent: "100x100")

# Vips style
blob.variant(resize_to_fill: [100, 100])
```
Common equivalents:

| MiniMagick | Vips |
|---|---|
| `resize: "WxH"` | `resize_to_fit: [W, H]` |
| `resize: "WxH^"` | `resize_to_fill: [W, H]` |
| `resize: "WxH>"` | `resize_to_limit: [W, H]` |
| `gravity: "Center", extent: "WxH"` | `resize_to_fill: [W, H]` |
| `strip: true` | `saver: { strip: true }` |
