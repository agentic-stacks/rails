# Hardening

Apply these settings to every production Rails app. Most are off by default or require explicit configuration.

---

## Content Security Policy

CSP tells browsers which sources are permitted for scripts, styles, images, and other resources. Configure it in the initializer generated by Rails:

```ruby
# config/initializers/content_security_policy.rb
Rails.application.config.content_security_policy do |policy|
  policy.default_src :self
  policy.font_src    :self, :https, :data
  policy.img_src     :self, :https, :data
  policy.object_src  :none
  policy.script_src  :self
  policy.style_src   :self

  # Turbo and importmap require :unsafe_inline for styles in some setups.
  # Prefer nonces instead:
  policy.script_src  :self, :nonce
  policy.style_src   :self, :nonce

  # Report violations without blocking (test mode):
  # policy.report_uri "/csp-violation-report"
end

# Use nonce-based scripts (required when using nonce above)
Rails.application.config.content_security_policy_nonce_generator =
  ->(_request) { SecureRandom.base64(16) }

Rails.application.config.content_security_policy_nonce_directives = %w[script-src style-src]
```

Use the nonce in views:

```erb
<%= javascript_tag nonce: true do %>
  console.log("This script is nonce-permitted.");
<% end %>
```

### Report-Only Mode

Enable report-only to observe violations before enforcing:

```ruby
Rails.application.config.content_security_policy_report_only = true
```

## CORS

Install rack-cors for API apps that are accessed from a browser on a different origin:

```ruby
# Gemfile
gem "rack-cors"
```

```ruby
# config/initializers/cors.rb
Rails.application.config.middleware.insert_before 0, Rack::Cors do
  allow do
    origins "https://app.example.com"  # never use "*" in production

    resource "/api/*",
      headers: :any,
      methods: [:get, :post, :put, :patch, :delete, :options],
      credentials: true,
      max_age: 600
  end
end
```

Never use `origins "*"` when `credentials: true` — browsers reject that combination, and it exposes session tokens to any origin.

## Rate Limiting

### Rails 8 rate_limit

Rails 8 adds `rate_limit` directly to controllers:

```ruby
class SessionsController < ApplicationController
  rate_limit to: 10, within: 3.minutes, only: :create,
             with: -> { render json: { error: "Too many requests" }, status: :too_many_requests }
end
```

### rack-attack (Rails 7 and older, or more complex rules)

```ruby
# Gemfile
gem "rack-attack"
```

```ruby
# config/initializers/rack_attack.rb
class Rack::Attack
  # Throttle login attempts by IP
  throttle("logins/ip", limit: 10, period: 1.minute) do |req|
    req.ip if req.path == "/sessions" && req.post?
  end

  # Throttle login attempts by email
  throttle("logins/email", limit: 5, period: 20.seconds) do |req|
    if req.path == "/sessions" && req.post?
      req.params["email"].to_s.downcase.gsub(/\s+/, "").presence
    end
  end

  # Blocklist known bad actors
  blocklist("block-bad-actors") do |req|
    BlockedIp.exists?(ip: req.ip)
  end

  # Return 429 with a plain response
  self.throttled_responder = lambda do |_env|
    [429, { "Content-Type" => "text/plain" }, ["Rate limit exceeded. Try again later."]]
  end
end
```

Mount the middleware:

```ruby
# config/application.rb
config.middleware.use Rack::Attack
```

## Mass Assignment

Strong parameters prevent mass assignment vulnerabilities. Always explicitly list permitted attributes.

```ruby
# Good — explicit permit
def user_params
  params.require(:user).permit(:name, :email, :password, :password_confirmation)
end

# Never do this — permits every attribute including role, admin, etc.
params.require(:user).permit!
```

For nested attributes:

```ruby
def post_params
  params.require(:post).permit(
    :title, :body,
    tags_attributes: [:id, :name, :_destroy],
    author_attributes: [:name, :email]
  )
end
```

## SQL Injection

Active Record parameterizes queries automatically. Bypass it only by mistake.

```ruby
# Safe
User.where("role = ?", params[:role])
User.where(role: params[:role])
User.where("name LIKE ?", "%#{ActiveRecord::Base.sanitize_sql_like(params[:q])}%")

# Unsafe — never do any of these
User.where("role = '#{params[:role]}'")
User.order(params[:column])              # order injection
User.find_by("id = #{params[:id]}")

# Safe dynamic column (whitelist first)
SORTABLE = %w[name created_at].freeze
column = SORTABLE.include?(params[:sort]) ? params[:sort] : "created_at"
User.order(column)
```

## XSS

ERB escapes all output by default. The only way to introduce XSS is to explicitly bypass the escaping.

```ruby
# Safe — escaped
<%= @user.bio %>

# Unsafe — marks string as HTML-safe without sanitizing
<%= @user.bio.html_safe %>
<%= raw @user.bio %>

# Safe — use sanitize when you must render HTML from users
<%= sanitize @user.bio, tags: %w[p b i a br], attributes: %w[href] %>
```

Never call `html_safe` or `raw` on user-supplied content. Use the `sanitize` helper with a strict allowlist.

For JSON embedded in HTML:

```erb
<!-- Unsafe — XSS via </script> in JSON -->
<script>var data = <%= @data.to_json %>;</script>

<!-- Safe -->
<script>var data = <%= json_escape(@data.to_json) %>;</script>
```

## CSRF

CSRF protection is on by default for all non-GET/HEAD/OPTIONS requests in HTML apps.

```ruby
# app/controllers/application_controller.rb
class ApplicationController < ActionController::Base
  protect_from_forgery with: :exception  # raises InvalidAuthenticityToken
end
```

For API controllers that skip sessions, disable CSRF and enforce token authentication:

```ruby
class Api::ApplicationController < ActionController::API
  # ActionController::API does not include CSRF protection by default
  before_action :authenticate_by_token
end
```

For mixed HTML/API apps:

```ruby
class Api::ApplicationController < ApplicationController
  skip_before_action :verify_authenticity_token
  before_action :authenticate_by_token
end
```

## Secure Headers

### Force SSL

```ruby
# config/environments/production.rb
config.force_ssl = true
```

This enables:
- HTTP → HTTPS redirect
- `Strict-Transport-Security` header (HSTS)
- Secure flag on session cookie

### X-Frame-Options

Already set to `SAMEORIGIN` by default. Override to block all framing:

```ruby
# config/application.rb
config.action_dispatch.default_headers["X-Frame-Options"] = "DENY"
```

### Full Secure Header Set

```ruby
# config/application.rb
config.action_dispatch.default_headers = {
  "X-Frame-Options"        => "SAMEORIGIN",
  "X-Content-Type-Options" => "nosniff",
  "X-XSS-Protection"       => "0",          # modern browsers: use CSP instead
  "Referrer-Policy"         => "strict-origin-when-cross-origin",
  "Permissions-Policy"      => "camera=(), microphone=(), geolocation=()"
}
```

## Secrets Audit

Store all secrets in Rails encrypted credentials or environment variables. Never commit plaintext secrets.

```bash
# Edit encrypted credentials
bin/rails credentials:edit

# Per-environment credentials (Rails 6+)
bin/rails credentials:edit --environment production
```

```yaml
# config/credentials.yml.enc
secret_key_base: "<generated>"
stripe:
  secret_key: "sk_live_..."
  webhook_secret: "whsec_..."
sendgrid_api_key: "SG...."
```

Access in code:

```ruby
Rails.application.credentials.stripe[:secret_key]
Rails.application.credentials.dig(:stripe, :secret_key)
```

Never use `ENV["SECRET"]` and `credentials` interchangeably in the same app — pick one pattern.

## Dependency Scanning

### bundle-audit

```ruby
# Gemfile (development/test group)
gem "bundler-audit", require: false
```

```bash
# Update advisory database and check
bundle exec bundle-audit check --update

# Ignore a specific advisory (with justification in a comment)
bundle exec bundle-audit check --ignore CVE-2023-XXXXX
```

### Brakeman

Brakeman performs static analysis and reports common Rails vulnerability patterns.

```ruby
# Gemfile (development group)
gem "brakeman", require: false
```

```bash
# Run a scan
bundle exec brakeman

# Output as JSON for CI parsing
bundle exec brakeman -f json -o brakeman-report.json

# Ignore a false positive (creates brakeman.ignore)
bundle exec brakeman -I

# Exit non-zero on any warning (useful for CI)
bundle exec brakeman --exit-on-warn
```

Categories Brakeman checks:
- SQL injection (string interpolation in queries)
- XSS (unescaped output, `html_safe` misuse)
- Mass assignment (`permit!`)
- Redirect safety (open redirect via user input)
- Command injection
- Unsafe deserialization

Run both tools in CI on every pull request. Treat new warnings as build failures.
