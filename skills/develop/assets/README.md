# Asset Pipeline: Decision Guide

Rails 7.x and 8.x support multiple asset pipeline configurations. The right choice depends on your JavaScript and CSS needs. This guide helps you pick the right combination.

## The Two Axes

Rails asset pipelines have two independent concerns:

1. **Asset serving** — how static files (JS, CSS, images) are fingerprinted and served. Options: Propshaft (Rails 8 default), Sprockets (Rails 7 default).
2. **JavaScript bundling** — how JS modules are assembled. Options: Import Maps (no bundler), jsbundling-rails (esbuild, rollup, webpack).
3. **CSS processing** — how CSS is compiled. Options: no processing (plain CSS via Propshaft), cssbundling-rails (Tailwind, PostCSS, Sass).

---

## Decision Table

| What You Need | Recommendation | Notes |
|---|---|---|
| Simple app, vanilla JS or Stimulus | **Import Maps + Propshaft** | Rails 8 default. Zero Node.js required. |
| React, Vue, or heavy npm ecosystem | **jsbundling (esbuild) + Propshaft** | Fast builds, tree shaking, TypeScript. |
| Tailwind CSS | **cssbundling (Tailwind) + Propshaft** | Or use the Tailwind CDN play version. |
| Sass / SCSS | **cssbundling (Sass) + Propshaft** | Sass requires Node. |
| Existing Sprockets app (Rails 7) | **Keep Sprockets or migrate to Propshaft** | See migration notes below. |
| API-only app | **None** | Skip asset pipeline entirely. |

---

## Rails 8 Defaults

A new Rails 8 app with no flags uses:

- **Propshaft** — asset serving and fingerprinting
- **Import Maps** — JavaScript modules, no bundler
- **Plain CSS** — no preprocessing unless you add cssbundling

```bash
rails new myapp                          # Propshaft + Import Maps + plain CSS
rails new myapp --css tailwind           # Propshaft + Import Maps + Tailwind
rails new myapp --javascript esbuild     # Propshaft + esbuild + plain CSS
rails new myapp --css tailwind \
                --javascript esbuild     # Propshaft + esbuild + Tailwind
```

---

## Rails 7 Defaults

A new Rails 7 app uses Sprockets for asset serving and Import Maps for JavaScript.

```bash
rails new myapp                          # Sprockets + Import Maps
rails new myapp --asset-pipeline propshaft   # Propshaft + Import Maps
rails new myapp --javascript esbuild         # Sprockets + esbuild
rails new myapp --css tailwind               # Sprockets + Tailwind
```

## Sprockets vs Propshaft

| Feature | Sprockets | Propshaft |
|---|---|---|
| Asset concatenation | Yes | No |
| ERB preprocessing in assets | Yes | No |
| Sass/CoffeeScript built-in | Yes | No |
| Fingerprinting | Yes | Yes |
| Simplicity | Complex | Simple |
| Performance | Slower | Faster |
| Recommended for new apps | No | Yes |

**Migrate from Sprockets to Propshaft** if:

- You are starting a new Rails 7+ app
- Your Sprockets config is minimal (no ERB in assets, no Sass via Sprockets)

Do not migrate if you rely on Sprockets-specific features like asset ERB preprocessing or the Sprockets JS concatenation pipeline.

### Migration Steps (Sprockets → Propshaft)

```bash
# 1. Remove Sprockets, add Propshaft
bundle remove sprockets sprockets-rails
bundle add propshaft

# 2. Delete config/initializers/assets.rb if present
# 3. Remove config.assets.* lines from config/application.rb and config/environments/
# 4. Ensure asset files don't use ERB (.js.erb, .css.erb for assets — not views)
# 5. Update asset paths — Propshaft serves from app/assets/, lib/assets/, vendor/assets/
bin/rails assets:precompile   # verify it works
```

---

## Asset Load Path

Propshaft and Sprockets both serve files from:

```
app/assets/images/
app/assets/stylesheets/
app/assets/javascripts/
lib/assets/
vendor/assets/
```

Gems can add their own asset paths. Files are served from `/assets/` and fingerprinted as `/assets/logo-abc123.png`.

---

## Production Build

```bash
# Precompile all assets for production
bin/rails assets:precompile

# Clean old compiled assets
bin/rails assets:clean

# Both (typical deploy step)
bin/rails assets:precompile && bin/rails assets:clean[2]
```

Output goes to `public/assets/`. Fingerprinted filenames are mapped in a manifest.

---

## Running in Development with bin/dev

When using jsbundling or cssbundling, `bin/rails server` alone is insufficient. Use `bin/dev` to run the Rails server and asset watchers together:

```bash
bin/dev   # Starts Procfile.dev processes
```

`Procfile.dev` is generated by jsbundling and cssbundling installers:

```
web: bin/rails server -p 3000
js: yarn build --watch
css: yarn build:css --watch
```

Import Maps + Propshaft apps do **not** need `bin/dev` — there is no build step.
